Three Integration Approaches

How to wire CodeMirror 6 and ProseMirror into the eg-walker CRDT pipeline

A

Dual Editor — CM6 (text) + PM (structural)

Two separate editor instances, each doing what it's best at. Maps directly to your existing split-view architecture.

┌─────────────────────┐    ┌─────────────────────────┐
│   CodeMirror 6      │    │     ProseMirror          │
│   (text pane)       │    │     (tree pane)          │
│                     │    │                           │
│ • syntax highlight  │    │ • custom AST schema       │
│ • line numbers      │    │ • block drag-drop         │
│ • code keybindings  │    │ • inline node editing     │
│ • multi-cursor      │    │ • collapse/expand         │
└────────┬────────────┘    └──────────┬────────────────┘
         │ ChangeSet                  │ Transaction (Steps)
         │ (char-level)              │ (structural)
         ▼                            ▼
┌─────────────────────────────────────────────────────┐
│              Editor Adapter Layer                     │
│  CM6 changes → insert/delete ops                     │
│  PM steps → TreeEditOp → unparse → text diff → ops  │
└────────────────────────┬────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────┐
│          eg-walker CRDT  (MoonBit / SyncEditor)      │
│  TextDoc ← FugueMax ← OpLog ← causal graph          │
│  Memo: text → parse → AST → ProjNode → SourceMap    │
└─────────────────────────────────────────────────────┘

Pros

  • Each editor used for its strength
  • Maps cleanly to existing text+tree architecture
  • Independent upgrade/replace of either pane

Cons

  • Two editor integrations to maintain
  • Synchronizing selection/focus between panes
  • Two dependency trees
B

ProseMirror Outer Shell + CM6 NodeViews

ProseMirror as the unified document model. Code-editing regions embed CM6 instances inside PM NodeViews. Recommended.

┌─────────────────────────────────────────────────────┐
│                  ProseMirror                         │
│          (unified document model)                    │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐                 │
│  │ TermNode      │  │ TermNode      │  ← PM nodes   │
│  │ (structural)  │  │ (structural)  │    (AST shape) │
│  │               │  │               │                │
│  │ ┌──────────┐ │  │ ┌──────────┐ │                 │
│  │ │ CM6 view │ │  │ │ CM6 view │ │  ← embedded     │
│  │ │ (inline) │ │  │ │ (inline) │ │    code editing  │
│  │ └──────────┘ │  │ └──────────┘ │                 │
│  └──────────────┘  └──────────────┘                 │
│                                                      │
│  Transaction (Steps) — both structural + text        │
└────────────────────────┬────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│              CRDT Bridge Layer                        │
│  PM Steps → classify:                                │
│    structural → TreeEditOp → tree_edit_bridge         │
│    text (from CM6 NodeView) → insert/delete ops      │
└────────────────────────┬────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────┐
│          eg-walker CRDT  (MoonBit / SyncEditor)      │
└─────────────────────────────────────────────────────┘

Pros

  • Single document model — PM schema mirrors your AST
  • Best of both: PM structural editing + CM6 text editing
  • Well-documented pattern (prosemirror.net/examples/codemirror/)
  • One selection system, one undo stack, one transaction flow
  • Natural fit for projectional editing framework

Cons

  • PM NodeView ↔ CM6 coordination is non-trivial
  • PM schema must faithfully represent your AST grammar
  • Harder to reason about: PM state + embedded CM6 states
C

CodeMirror 6 Only + Structural Decorations

CM6 as the single editor. Use decorations, widgets, and gutter markers to overlay structural editing features on top of text.

┌─────────────────────────────────────────────────────┐
│                  CodeMirror 6                         │
│           (text model + decorations)                 │
│                                                      │
│  λx. ───────── [collapse widget]                     │
│    let y = 1 ─ [type annotation decoration]          │
│    x + y ───── [AST node highlight]                  │
│                                                      │
│  Structural ops via:                                 │
│  • Fold/unfold (native CM6 folding)                  │
│  • Widget decorations (inline controls)              │
│  • Gutter markers (node type indicators)             │
│  • Keybindings (structural transforms)               │
└────────────────────────┬────────────────────────────┘
                         │ ChangeSet (always char-level)
                         ▼
┌─────────────────────────────────────────────────────┐
│          eg-walker CRDT  (MoonBit / SyncEditor)      │
└─────────────────────────────────────────────────────┘

Pros

  • Simplest integration — one editor, one model
  • Direct char-level mapping to CRDT (no translation)
  • Lightest weight, fastest to build MVP

Cons

  • Structural editing is bolted-on, not native
  • Limited: can't do true block drag-drop, node insertion
  • Doesn't scale well as a general projectional editing framework
  • Fighting CM6's linear text model for tree operations

My recommendation: Approach B — ProseMirror as the outer shell with CM6 NodeViews for code regions. It's the natural architecture for a projectional editing framework: PM's schema-driven document model is a projection of your AST, and CM6 handles the text editing UX where you need it. This pattern is proven (Notion-style editors use it) and scales to new content types by defining new PM schemas.
