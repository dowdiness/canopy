ProseMirror vs Tiptap

Tiptap is built on ProseMirror — the question is whether the abstraction layer helps or hinders your use case

ProseMirror

What it is

Low-level toolkit. You compose EditorState, EditorView, Schema, Plugins, and Transactions yourself. No opinions — maximum control.

Pros

  • Full control over document model — you define the schema, transaction handling, and state management exactly how you need
  • Custom CRDT integration — you can intercept transactions at any level and route them to your eg-walker CRDT instead of ProseMirror's built-in history
  • No abstraction leaks — when something breaks, you're debugging ProseMirror directly, not Tiptap-wrapping-ProseMirror
  • Framework-agnostic — vanilla JS. Works with React, MoonBit/Rabbita, or no framework
  • Battle-tested collab — prosemirror-collab is the reference implementation for collaborative editing
  • Mature & stable API — rarely breaking changes since 1.0 (2018)
  • Perfect for framework builders — you're building a general projectional editing framework, not an app. PM gives you the primitives.

Cons

  • Steep learning curve — you must understand EditorState, Transactions, Plugins, NodeViews, Decorations, Schema spec
  • Verbose setup — basic editor requires ~50-100 lines of boilerplate
  • No built-in UI — menus, toolbars, drag handles are all DIY
  • Documentation is dense — comprehensive but assumes deep knowledge
Tiptap

What it is

Opinionated wrapper around ProseMirror. Provides Extension API, React/Vue bindings, and pre-built components. Open-core model (some features paid).

Pros

  • Faster to prototype — get a working editor in ~10 lines
  • Extension system — StarterKit bundles common extensions (bold, italic, headings, etc.)
  • React/Vue first-class — if you're using React, hooks and components are ready
  • Pre-built collab — Hocuspocus server + Yjs integration out of box
  • Better docs for beginners — guides, examples, interactive playground

Cons

  • Abstraction over your CRDT — Tiptap's collab assumes Yjs. Replacing with eg-walker means fighting the abstraction, not benefiting from it.
  • Open-core model — drag handle, AI, advanced collab features are paid (Tiptap Cloud). Your framework would depend on a commercial product's decisions.
  • Extra indirection — debugging goes through Tiptap → ProseMirror → DOM. You'll end up reading PM source anyway.
  • Framework coupling — Tiptap's React/Vue bindings assume those frameworks. Your MoonBit/Rabbita frontend would bypass them entirely.
  • Upgrade risk — Tiptap has had breaking changes between major versions. You'd track both Tiptap AND ProseMirror versions.
  • Wrong abstraction level for framework builders — Tiptap is designed for app developers, not for people building editor frameworks. It hides exactly the parts you need to control.

For your specific case

→

ProseMirror is the better fit

You're building a general projectional editing framework with a custom CRDT (eg-walker, not Yjs). Tiptap's value proposition — quick setup, pre-built collab, React bindings — doesn't apply because:

  • You'd replace Tiptap's collab layer with eg-walker anyway
  • Your frontend is MoonBit/Rabbita, not React/Vue
  • You need deep control over the document model for AST↔text bidirectional sync
  • You're building the framework that Tiptap-like wrappers would be built on top of
