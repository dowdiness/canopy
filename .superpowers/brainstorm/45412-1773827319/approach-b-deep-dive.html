Approach B Deep Dive: ProseMirror Shell + CM6 NodeViews

How ProseMirror's schema maps to your AST, and how edits flow through the CRDT

1. PM Schema ↔ AST Term Mapping

Your @ast.Term variants become ProseMirror node types. PM's schema system enforces valid AST structure.

Schema Mapping: Term → PM NodeSpec
MoonBit Term PM Node Type Content Expression Editable Part
Int(n) int_literal (leaf — inline text) CM6 inline: number
Var(name) var_ref (leaf — inline text) CM6 inline: identifier
Lam(x, body) lambda "param term" Param name: CM6 inline.
Body: nested PM node.
App(f, x) application "term term" Both children: nested PM nodes.
Bop(op, l, r) binary_op "term term" Operator: attr. Operands: nested PM nodes.
If(c, t, e) if_expr "term term term" 3 children: nested PM nodes.
Module(defs, body) module "let_def* term" Top-level container. Defs + final expr.
Error(msg) error_node (leaf) Red highlight, shows message. Not editable.

3. CM6 NodeView: Where Text Editing Happens

Leaf nodes and editable names use embedded CodeMirror 6 instances inside ProseMirror NodeViews.

What the user sees
let double = λx. x + x
double 5

Blue-highlighted spans = CM6 inline editors (click to edit)
Structure between them = PM nodes (drag to restructure)

PM document tree (internal)
module
├─ let_def {name: "double"}
│  └─ lambda {param: "x"}
│     └─ binary_op {op: "Plus"}
│        ├─ var_ref {name: "x"}  ← CM6
│        └─ var_ref {name: "x"}  ← CM6
└─ application
   ├─ var_ref {name: "double"}   ← CM6
   └─ int_literal {value: 5}     ← CM6

4. Edit Flow Examples

Flow A: User types in a CM6 NodeView (text edit)
User renames variable "x" → "y" inside CM6 NodeView
  │
  ├─ 1. CM6 onChange fires with ChangeSet {from: 0, to: 1, insert: "y"}
  │
  ├─ 2. CM6 NodeView.forwardUpdate() →
  │      converts CM6 offset to PM document position →
  │      dispatches PM Transaction: ReplaceStep(pos, pos+1, "y")
  │
  ├─ 3. Bridge classifies: leaf text edit →
  │      calls SyncEditor.insert_and_record() / delete_and_record()
  │      (character-level CRDT ops, NOT set_text)
  │
  ├─ 4. CRDT generates ops → broadcast via export_since()
  │
  └─ 5. Memo chain: text → re-parse → new ProjNode (stable IDs) → SourceMap
         PM doc stays in sync (originated here, no reconcile needed)
Flow B: User drags a node (structural edit)
User drags "double 5" application above the let_def
  │
  ├─ 1. PM drag plugin fires → produces ReplaceStep (move content)
  │
  ├─ 2. Bridge classifies: structural edit →
  │      maps PM node positions to NodeIds (via SourceMap) →
  │      constructs TreeEditOp::Drop(source, target, position)
  │
  ├─ 3. SyncEditor.apply_tree_edit(op, timestamp) →
  │      apply_edit_to_proj() → modified ProjNode →
  │      unparse → new text → set_text_and_record() →
  │      character-level CRDT diff ops generated
  │
  ├─ 4. CRDT ops broadcast
  │
  └─ 5. Reconciler: new ProjNode → diff against PM doc → PM Transaction
         Updates PM doc to match new structure
Flow C: Remote peer sends changes
WebSocket receives SyncMessage from peer
  │
  ├─ 1. apply_sync_json() → CRDT merges remote ops
  │
  ├─ 2. Memo chain re-derives: text → ProjNode → SourceMap
  │
  ├─ 3. Reconciler diffs old vs new ProjNode →
  │      generates PM Transaction with ReplaceSteps →
  │      dispatches with {remote: true} metadata
  │
  └─ 4. PM applies transaction → NodeViews update →
         CM6 instances receive new content via update() method
         (updating flag prevents feedback loop)

5. Key Design Decision: Where is the Source of Truth?

Authority Model
  ╔════════════════════════════════════════════════════════════════╗
  ║  CRDT TextDoc is ALWAYS the source of truth                   ║
  ║                                                                ║
  ║  PM EditorState is a DERIVED VIEW — like a React component    ║
  ║  rendering from a store. It can propose changes, but the      ║
  ║  CRDT decides the canonical state.                            ║
  ║                                                                ║
  ║  This means:                                                   ║
  ║  • PM state may temporarily diverge during edit → reconcile   ║
  ║  • Undo/redo uses SyncEditor.UndoManager, not PM history      ║
  ║  • Collab uses eg-walker sync, not prosemirror-collab          ║
  ║  • PM schema validation is advisory (CRDT text may parse to   ║
  ║    Error nodes — PM renders them, doesn't reject them)        ║
  ╚════════════════════════════════════════════════════════════════╝

6. Framework Generalization Path

How this architecture extends beyond lambda calculus:

Pluggable Schema Layer
┌──────────────────────────────────────────────────────────┐
│                  Schema Registry                          │
│                                                           │
│  register("lambda-calculus", {                           │
│    astToSchema:  Term → PM NodeSpec mapping               │
│    schemaToAst:  PM Node → Term mapping                   │
│    parser:       text → CST → AST                         │
│    printer:      AST → text                               │
│    lezerGrammar: (optional) for CM6 syntax highlighting   │
│  })                                                       │
│                                                           │
│  register("markdown", { ... })     ← future              │
│  register("json-schema", { ... })  ← future              │
│  register("custom-dsl", { ... })   ← user-defined        │
└──────────────────────────────────────────────────────────┘

The Bridge Layer stays the same — it works with:
  • PM Transactions (generic)
  • TreeEditOps (generic structural ops)
  • SyncEditor (generic text CRDT)

Only the Schema mapping is language-specific.
