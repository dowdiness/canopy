# Edit Layer Error Model

## Why

The edit layer reports expected edit failures as strings. The shared contract in
`lang/runtime/language_spec.mbt` carries `Result[..., String]`, lambda defines
`EditResult = Result[(Array[SpanEdit], FocusHint)?, String]`, JSON and Markdown
compute edits return the same shape, and companion/FFI adapters flatten those
strings again at the boundary.

This loses structure just before the editor boundary. `editor/errors.mbt`
already has a typed `TreeEditError` with `message()`, but
`lang/lambda/companion/lambda_editor.mbt` currently maps language edit strings
to `TreeEditError::ProjectionEdit(detail=msg)`, and JSON/Markdown expose raw
strings through their companion APIs. PR #383 finding D and issue #667 both
deferred this cleanup because a raising error model can remove the repeated
`Ok(v) => v; Err(e) => return Err(e)` passthroughs and delete lambda's
zero-information `ResolveError`.

## Scope

In:
- `core/` for the shared `EditError` type and its message renderer.
- `lang/runtime/language_spec.mbt` and `lang/runtime/language_spec_test.mbt`.
- `lang/lambda/edits/*.mbt` and affected `*_wbtest.mbt`.
- `lang/lambda/companion/lambda_editor.mbt` and tree-edit FFI callers that
  render `TreeEditError::message()`.
- `lang/json/edits/compute_json_edit.mbt`, `lang/json/companion/json_companion.mbt`,
  `ffi/json/edit.mbt`, and affected tests.
- `lang/markdown/edits/compute_markdown_edit.mbt`,
  `lang/markdown/edits/compute_move_block.mbt`,
  `lang/markdown/companion/markdown_companion.mbt`,
  `ffi/markdown/markdown_ffi.mbt`, and affected tests.
- `editor/errors.mbt` if typed language-edit mapping needs one or two new
  boundary variants.
- `.mbti` files generated by `moon info`.
- `docs/TODO.md` only to mark §7 finding D / #667 residue complete after the
  migration lands.

Out:
- `relay/`, `ephemeral/`, and `protocol/` error handling.
- Redesigning `editor/errors.mbt` beyond the edit-error mapping required here.
- Converting unrelated `abort` or `fail` sites.
- Changing JSON edit-op parsing in `ffi/json/edit.mbt` from string errors unless
  it is necessary to keep the public apply-edit path compiling.
- Changing external JS-visible error formats beyond preserving today’s string
  messages via `message()`-style rendering.

## Current State

- `lang/runtime/language_spec.mbt` defines the language-agnostic SPI with
  `compute_edit` and `on_no_edit` returning `Result[..., String]`; its
  `apply_edit` also returns `Result[Unit, String]` and manually passes
  `Err(msg)` through.
- A production grep across the planned surface finds 52 `Result[..., String]`
  signature sites, including 20 in `lang/lambda/edits`. Tests add additional
  stub and assertion churn.
- `lang/lambda/edits/text_edit.mbt` defines `EditResult` as a `String` result
  and a private `ResolveError(String)` used only by `EditContext::resolve`.
  There are 13 `ResolveError(e) => return Err(e)` catch arms in
  `text_edit_commit.mbt`, `text_edit_delete.mbt`, `text_edit_drop.mbt`,
  `text_edit_refactor.mbt`, `text_edit_structural.mbt`, and
  `text_edit_wrap.mbt`.
- Lambda error strings cluster around:
  - node/source-map/registry lookup failures, e.g. `Node not found: ...`,
    `Parent not found: ...`, `Binding not found: ...`, `Module not found: ...`;
  - invalid targets or unsupported operations, e.g. `AddBinding target is not a
    Module node`, `SwapChildren requires Bop or If node`, `ChangeOperator
    requires Bop node`, `InlineDefinition requires a Var node`;
  - scope and capture failures, e.g. `Cannot rename: would capture free
    variable ...`, `Cannot move down: ... uses ...`, `Cannot inline: would
    capture variables bound by enclosing lambda`;
  - source-span recovery failures, e.g. `Cannot rewrite fn binding: ... range
    not found`, `Cannot inline binding: init source range not found`;
  - edit invariants, e.g. `Cannot drop onto self`, `Cannot drop into own
    descendant`, `Child index out of range`, `Negative index`.
- JSON edit errors in `lang/json/edits/compute_json_edit.mbt` cluster around
  missing source-map/projection nodes, malformed structural targets such as
  missing closing braces/brackets, invalid unwrap child count, and missing key
  spans.
- Markdown edit errors in `lang/markdown/edits` cluster around missing
  source-map/projection nodes, missing editable/text spans, unsupported block
  moves, list/container move restrictions, unparseable ordered-list markers,
  loose-list rejection, and unclosed fenced-code rejection.
- `editor/errors.mbt` already defines typed `TreeEditError` with `message()`;
  its current string escape hatches are `TextTransformFailed(detail)` and
  `ProjectionEdit(detail)`.
- `lang/lambda/companion/lambda_editor.mbt` converts
  `compute_text_edit(...).Err(msg)` to `TreeEditError::ProjectionEdit(detail=msg)`.
  `ffi/lambda/intent.mbt` and `ffi/lambda/semantic.mbt` expose those failures as
  `"error: " + err.message()`.
- `ffi/json/edit.mbt` exposes JSON edit failures as `"error: " + err` and logs
  the stripped message in `JsonEditLogEntry`.
- `ffi/markdown/markdown_ffi.mbt` exposes Markdown edit failures in a JSON
  object as `"message": msg`.
- Dependency direction matters: `lang/runtime/moon.pkg` imports `editor`.
  Therefore `editor` cannot import an `EditError` defined in `lang/runtime`
  without creating a cycle. `core/moon.pkg` is already imported by `editor`,
  `lang/runtime`, and all three language edit packages, so `core` is the shared
  home for `EditError` unless implementation uncovers an unexpected MoonBit
  packaging constraint.

## Desired State

- Expected language edit failures use a typed `pub(all) suberror EditError`,
  likely in `core`, with a stable `EditError::message() -> String` renderer.
- Language edit APIs use raising signatures, for example
  `compute_* -> (Array[SpanEdit], FocusHint)? raise EditError`, so ordinary
  propagation is zero-syntax and manual `Result` passthroughs disappear.
- `lang/runtime.LanguageSpec` expresses the shared contract in terms of
  `raise EditError`; JSON, Markdown, and lambda implementations conform to that
  contract.
- Editor-facing adapters catch `EditError` only at language/editor or FFI
  boundaries and map it structurally into `TreeEditError` or strings.
- `ResolveError` is deleted and its 13 catch arms are gone.
- JS-visible behavior remains string-based at the outer FFI boundary, with the
  same broad message text unless a test is deliberately updated for a clearer
  typed renderer.

## Steps

1. **Add the shared `EditError` type in `core`.**
   Define `pub(all) suberror EditError` and `EditError::message() -> String`
   in a new focused `core` file. Use semantic variants derived from the current
   strings rather than one variant per string: `ProjectionUnavailable`,
   `NodeNotFound`, `NodeNotInSourceMap`, `NodeNotInRegistry`,
   `BindingNotFound`, `InvalidTarget`, `UnsupportedOperation`,
   `CaptureViolation`, `ScopeViolation`, `ParseFailed`, `SourceSpanMissing`,
   and `Invariant(detail)`. Keep enough fields for mapping and messages
   (`action`, `node_id`, `role`, `operation`, `detail`) without depending on
   language-specific AST types. This step should compile because no callers use
   it yet.

   Validation:
   ```bash
   moon check
   moon test core
   ```

   Invariants: `core` remains below `editor`, `lang/runtime`, and language
   packages in the dependency graph; `EditError::message()` can reproduce the
   legacy user-facing messages for existing tests.

2. **Convert lambda leaf helpers and `ResolveError` to typed raises.**
   Migrate `EditContext::resolve` to raise `@core.EditError` directly, then
   convert leaf helpers in `lang/lambda/edits/scope.mbt`,
   `text_edit_module_view.mbt`, `text_edit_utils.mbt`,
   `text_edit_binding_source.mbt`, and the range/reference helpers in
   `text_edit_rename.mbt`. Replace string construction with the taxonomy from
   step 1 and keep the renderer compatible with the old text. Delete
   `ResolveError` after the direct callers are converted.

   **Compilation break point:** while a helper signature has changed but its
   callers still expect `Result[..., String]`, `moon check` will fail. Fix
   sequence: convert the helper signature, convert its direct callers in the
   same file/package slice to `raise EditError`, then remove the obsolete
   `Err(e) => return Err(e)` arms before checking.

   Validation:
   ```bash
   moon check
   moon test lang/lambda/edits
   ```

   Invariants: all 13 `ResolveError` catch arms are gone; lookup failures still
   render as the same `Node not found: ...` / binding-not-found messages unless
   tests are intentionally updated to assert variants.

3. **Convert lambda edit dispatch to raising `EditResult`.**
   Change lambda’s `EditResult` alias from `Result[(Array[SpanEdit],
   FocusHint)?, String]` to the bare optional edit payload returned by functions
   that `raise @core.EditError`. Migrate `core_dispatch`, `compute_text_edit`,
   and every lambda handler (`commit`, `delete`, `drop`, `wrap`, `structural`,
   `binding`, `refactor`, `rename`) bottom-up. Keep `Ok(None)` semantics as
   plain `None`; keep no-op successes as `Some(([], RestoreCursor))`.

   **Compilation break point:** this is the lambda-wide call-shape flip.
   Minimize breakage by converting private handlers before public
   `compute_text_edit`, then immediately update `lambda_editor.mbt` in the next
   step. Do not stop between the public signature flip and the companion update.

   Validation:
   ```bash
   moon check
   moon test lang/lambda/edits
   ```

   Invariants: no lambda edit handler returns `Result[..., String]`; repeated
   passthrough matches are gone; behavior tests still cover success/no-op/error
   branches.

4. **Map lambda `EditError` at the editor boundary.**
   Update `lang/lambda/companion/lambda_editor.mbt` so `apply_lambda_tree_edit`
   catches `@core.EditError` from `compute_text_edit` and maps it to
   `@editor.TreeEditError` structurally. Prefer existing
   `ProjectionUnavailable`, `UnhandledOperation`, `NodeNotInSourceMap`, and
   `NodeNotInRegistry` where they fit; add at most one or two editor variants
   only if `CaptureViolation`/`ScopeViolation` need to stay distinguishable past
   the language boundary. Preserve `ffi/lambda/intent.mbt` and
   `ffi/lambda/semantic.mbt` as string renderers through `err.message()`.

   Validation:
   ```bash
   moon check
   moon test lang/lambda/companion
   moon test ffi/lambda
   ```

   Invariants: lambda FFI still returns `ok` or `error: <message>`; structure is
   available inside `TreeEditError`; old `ProjectionEdit(detail=msg)` is no
   longer the default language-error path.

5. **Convert `lang/runtime.LanguageSpec` to `raise EditError`.**
   Update `compute_edit`, `on_no_edit`, and `LanguageSpec::apply_edit` to use
   raising functions and return/catch at the appropriate boundary. Preserve the
   `None` no-edit routing: `compute_edit` returning `None` calls `on_no_edit`;
   JSON’s `on_no_edit` raises `UnsupportedOperation`/`UnhandledOperation`, while
   Markdown no-edit remains success. Convert `language_spec_test.mbt` stubs to
   typed raises and variant/message assertions.

   **Compilation break point:** JSON companion depends on this contract before
   JSON edits are migrated. Fix sequence: update `LanguageSpec`, update its
   tests, then immediately migrate JSON edits and companion in step 6 before
   running a workspace check.

   Validation:
   ```bash
   moon check
   moon test lang/runtime
   ```

   Invariants: no `Result[Unit, String]` remains in `lang/runtime`; the generic
   contract is the source of truth for typed edit failures.

6. **Migrate JSON edits, companion, and FFI rendering.**
   Convert `lang/json/edits/compute_json_edit.mbt` and its helpers to raise
   `@core.EditError`. Map current strings to variants: source-map/projection
   lookups to node lookup variants, closing-brace/bracket failures and invalid
   unwrap child count to `Invariant(detail)` or `InvalidTarget`, key-span
   failures to `SourceSpanMissing`, and unhandled no-edit to
   `UnsupportedOperation`. Update `lang/json/companion/json_companion.mbt` to
   match the runtime contract. In `ffi/json/edit.mbt`, catch/render
   `EditError::message()` so `json_apply_edit` and edit-log strings remain
   stable.

   Validation:
   ```bash
   moon check
   moon test lang/json/edits
   moon test lang/json/companion
   moon test ffi/json
   ```

   Invariants: JSON success edits are unchanged; failures still surface as
   `"error: <message>"`; `JsonEditLogEntry.error` continues storing the message
   without the `"error: "` prefix.

7. **Migrate Markdown edits, companion, and FFI rendering.**
   Convert `lang/markdown/edits/compute_markdown_edit.mbt` and
   `compute_move_block.mbt` to raise `@core.EditError`. Map source-map and
   projection lookup failures to lookup variants, missing editable/text spans
   and sibling separator spans to `SourceSpanMissing`, unsupported move modes
   and list/container restrictions to `UnsupportedOperation`, ordered-marker
   parse failures to `ParseFailed`, and unclosed fenced-code / loose-list
   blockers to `InvalidTarget` or `Invariant(detail)` with the legacy detail.
   Update `lang/markdown/companion/markdown_companion.mbt` to catch/render only
   at its FFI-facing boundary and preserve Markdown’s `None` no-op semantics.
   Update `ffi/markdown/markdown_ffi.mbt` to keep JSON error responses shaped
   as `{ status: "error", message: <string> }`.

   Validation:
   ```bash
   moon check
   moon test lang/markdown/edits
   moon test lang/markdown/companion
   moon test ffi/markdown
   ```

   Invariants: Markdown no-op behavior for first-block merge and adjacent moves
   is unchanged; snapshot churn is limited to tests that intentionally switch
   from raw `Err("...")` inspection to variant/message inspection.

8. **Clean tests, interfaces, and TODO state.**
   Replace direct `Err(msg)` assertions in lambda, JSON, Markdown, runtime, and
   companion tests with either `try/catch` variant checks or `message()` checks
   when the public contract is string rendering. Run interface generation and
   inspect `.mbti` changes carefully. Remove any remaining production
   `Result[..., String]` sites in the edit/runtime/companion surface unless
   they are explicitly out of scope, then update `docs/TODO.md` §7 finding D to
   reference this migration and issue #667 cleanup.

   Validation:
   ```bash
   moon check
   moon test
   moon info
   moon fmt
   git diff --stat
   git diff -- '*.mbti'
   ```

   Invariants: `.mbti` changes reflect intentional API moves from string
   results to typed raises; no unrelated formatting or source churn appears.

      `lang/markdown/edits`, and from `lang/runtime/language_spec.mbt`.

## Risks

- **Dependency cycle risk:** defining `EditError` in `lang/runtime` would make
  editor-side typed mapping impossible because `lang/runtime` already imports
  `editor`. Use `core` unless package inspection during implementation finds a
  better acyclic shared home.
- **Snapshot churn:** tests that inspect `Err("...")` will need conversion to
  `catch`/variant checks or `message()` checks. Keep renderer output stable to
  avoid user-visible churn.
- **`.mbti` surface changes:** moving public APIs from `Result[..., String]` to
  `raise EditError` intentionally changes interfaces. Inspect for accidental
  trait-bound widening after `moon info`.
- **Boundary flattening regression:** JSON, Markdown, and lambda JS callers
  still expect string errors. Every FFI path must render via `message()` at the
  outer boundary.
- **Over-broad taxonomy:** too many variants recreates stringly typing with enum
  names; too few variants collapses structure. Keep semantic variants grouped by
  catch-site needs and use `Invariant(detail)` as the escape hatch.
