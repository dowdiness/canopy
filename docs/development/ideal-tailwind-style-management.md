# Ideal Tailwind Style Management

This guide records the Tailwind v4 conventions for `examples/ideal/`. The model
is **shadcn-style source ownership**, not shadcn/ui adoption: UI recipes are local
project code authored in MoonBit, with Tailwind utilities generated by Ideal's
Vite build.

## Scope

- Applies to `examples/ideal/` only.
- Keep recipes Ideal-local until at least two in-repo apps need the same
  primitive and a shared package boundary is designed.
- Do not import shadcn/ui React components.
- Do not add Radix or another headless UI dependency unless an existing Rabbita
  or local headless model cannot cover a concrete behavior gap.

## Ownership Boundaries

| Area | Owns |
| --- | --- |
| `examples/ideal/main/view_*.mbt` | semantic structure, state-to-view composition, messages |
| `examples/ideal/main/*_classes.mbt` | slice-local static Tailwind utility bundles |
| `examples/ideal/main/ui/*.mbt` | reusable Ideal UI recipes such as buttons, inputs, panels, menu rows |
| `examples/ideal/web/styles/editor.css` | Tailwind imports, `@source`, `@theme inline` token bridge, global light-DOM rules, shared keyframes |
| `examples/ideal/web/styles/editor-shadow.css` | shadow-root rules delivered by `adoptedStyleSheets` |
| Rabbita/headless models | behavior, focus management, roving focus, subscriptions, commands |
| `protocol/`, `editor/`, `projection/` | framework data flow; no Tailwind classes |

A migrated visual declaration has exactly one live owner. When a declaration
moves to Tailwind utilities, delete the equivalent legacy CSS rule in the same
slice. Ownership includes cascade-layer ownership: Tailwind utility classes must
not be shadowed by later unlayered CSS rules for the same declaration.

## Cascade Layer Rules

Tailwind utilities are imported into `@layer utilities`, so unlayered custom CSS
can override migrated utility classes even with lower selector specificity. Keep
global resets and base primitives in `@layer base` when Tailwind utilities own
migrated spacing, color, border, or layout declarations:

```css
@layer base {
  * { margin: 0; padding: 0; box-sizing: border-box; }
}
```

Do not leave broad unlayered rules such as `* { padding: 0; }` next to migrated
Tailwind spacing utilities like `px-canopy-lg` or `mr-canopy-sm`. Keep global
focus-ring defaults in `@layer base` as well so slice-owned `focus-visible:*`
utilities can override outline geometry when needed.

## Class Recipe Rules

1. **Semantic hook first.** Every public class bundle starts with the stable hook
   used by tests and integration code:
   ```moonbit
   let action_overlay_panel_class = "action-overlay-panel fixed z-[51] ..."
   ```
2. **Static tokens only.** Tailwind must see utility tokens in source. Avoid
   runtime-generated fragments such as `"bg-" + tone`.
3. **Typed variants.** Use MoonBit enums/functions for variants instead of
   scattering string conditionals through views.
4. **Prefer tokens over literals.** Repeated color, spacing, radius, font, or
   motion values should become `@theme inline` aliases backed by existing
   `--canopy-*`, `--space-*`, or `--text-*` custom properties.
5. **Extract MoonBit constants for repetition.** If utility bundles get noisy,
   split them into named MoonBit constants or recipe helpers.
6. **Never use Tailwind `@apply`.** It is an antipattern for this repo. Do not
   introduce CSS component wrappers that re-hide utility ownership.

## Recipe Shape

A small Ideal UI recipe should expose typed class helpers, and only add view
helpers when doing so reduces repeated Rabbita markup without owning behavior.
Keep variant enums private unless callers genuinely need to choose among them;
public helpers should usually describe semantic intent, not implementation
variants.

```moonbit
priv enum ButtonTone {
  Accent
  Ghost
  Danger
}

priv enum ButtonSize {
  Small
  Medium
}

let button_base_class = "ideal-button inline-flex items-center ..."

fn button_tone_class(tone : ButtonTone) -> String {
  match tone {
    Accent => "bg-canopy-accent text-canopy-fg-inverse"
    Ghost => "bg-transparent text-canopy-secondary hover:bg-[rgba(130,80,223,0.15)]"
    Danger => "text-canopy-error-text hover:bg-canopy-error-bg"
  }
}

fn button_size_class(size : ButtonSize) -> String {
  match size {
    Small => "h-[28px] px-canopy-sm text-canopy-label"
    Medium => "h-[36px] px-canopy-md text-canopy-small"
  }
}

fn button_recipe(hook : String, tone : ButtonTone, size : ButtonSize) -> String {
  hook +
  " " +
  button_base_class +
  " " +
  button_tone_class(tone) +
  " " +
  button_size_class(size)
}

pub fn toolbar_button_class() -> String {
  button_recipe("toolbar-btn", Ghost, Small)
}

pub fn action_button_danger_class() -> String {
  button_recipe("action-btn danger", Danger, Medium)
}
```

Keep semantic hooks (`toolbar-btn`, `action-btn danger`, `ideal-button` above)
stable even if the utility bundle changes.

## Tailwind Source Rules

Keep source detection explicit and narrow. Add each migrated file intentionally
from `examples/ideal/web/styles/editor.css`:

```css
@source "../../main/view_actions_classes.mbt";
@source "../../main/ui/button.mbt";
```

Do not broaden to `../../main/**/*.mbt` until several slices prove that the
migration needs it and CSS size remains controlled.

## Testing Rules

Each migrated slice needs both behavior and style coverage:

- keep existing semantic selector tests green;
- add computed-style Playwright assertions for representative migrated
  declarations;
- include focus, active, hover, danger/error, and mobile/touch states when the
  slice owns them;
- compare built CSS size and investigate disproportionate growth.

## Next Slice Guidance

Prefer light-DOM slices before touching shadow-owned structure styles. The
action overlay/name prompt and toolbar/action-button chrome have already proven
narrow Tailwind scanning plus Ideal-local recipes. A good next target should
reuse those recipes or reveal one concrete missing primitive (for example an
input, panel, or menu-row recipe) without broad scanning.

Defer:

- shadow-owned structure styles;
- raw HTML fragments in history/incr/bottom panels;
- cross-app `lib/canopy-ui` extraction;
- broad Tailwind scanning;
- new UI dependencies.
